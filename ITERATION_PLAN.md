| 维度 | 状态 |
|------|------|
| 技术栈 | Vue 3 + TypeScript + Pinia + PixiJS v7 + Vite |
| 代码规模 | 51 个源文件，~3,500 行代码 |

## 二、问题清单

### 2.1 Bug（需优先修复）

| # | 问题描述 | 位置 | 严重性 |
|---|---------|------|--------|
| B1 | **停止仿真后统计数据被清除**：`simulationStore.stop()` 在第 248-253 行直接重置了 `tickCount`、`palletStates`、`zoneStates`、`forkliftCooldowns`、`multiRunResults`，导致用户停止仿真后无法查看分析结果。虽然代码注释说"保留分析数据供查看"，但实际逻辑矛盾。 | `simulationStore.ts:248-257` | **P0** |
| B2 | **消费者叉车 tick 时序问题**：当托盘到达输送机末端且下游为消费者时，`Simulation.tick()` 第 290 行先调用 `downstream.model.tick(dt)` 尝试消费，但此时消费者的冷却可能还未就绪，导致托盘被阻塞但消费者已进入新一轮冷却。应在托盘到达前就让消费者独立 tick。 | `Simulation.ts:288-304` | **P1** |
| B3 | **输送机末端阻塞后 zone 状态不一致**：当下游满导致托盘留在末端 zone 时（第 270-277 行），先 `removePallet(pallet)` 清除了 zone 占用，然后又手动设置 `conveyor.zones[exitZone].occupied = true`。但 `removePallet` 已将 pallet 从 `conveyor.pallets` 数组移除，随后又 `conveyor.pallets.push(pallet)` 放回，导致数组顺序可能乱序，影响后续 ZPA 排序。 | `Simulation.ts:262-277` | **P1** |
| B4 | **撤销栈在 stop 后未清理**：`canvasStore.undo()` 撤销后调用 `loadFromJSON`，但不刷新画布上已显示的连接线（需要手动触发 `refreshConveyors`）。虽然 `CanvasContainer` 有 watch 监听 `version`，但 `loadFromJSON` 确实会 `version++`，理论上可以触发。实际需验证在仿真中撤销后 canvas 状态是否正确同步。 | `canvasStore.ts:212-218` | **P2** |
| B5 | **叉车端口硬编码偏移**：`SnapManager.ts:69` 中叉车端口位置硬编码 `x + 26`，`ForkliftGraphic` 中叉车宽度为 52px，端口固定在右侧。但叉车旋转后端口位置不会随之变化，因为 `SelectionManager` 不支持叉车旋转（`getComponentData` 返回 `rotation: 0`）。 | `SnapManager.ts:69`, `SelectionManager.ts:211` | **P2** |
| B6 | **路由表 action 值与端口名不匹配**：路由表中 action 可选值为 `'straight'` 和 `'turn'`，但 `SnapManager` 中移载机的端口名是 `'north'`/`'south'`/`'east'`/`'west'`。`TransferModel` 第 45 行使用 `routingTable[pallet.destination] ?? defaultRoute`，返回的 `'straight'`/`'turn'` 需要映射为实际端口名，但代码中没有映射逻辑。 | `TransferModel.ts:45`, `RoutingTable.vue` | **P1** |
| B7 | **双向输送机逆流托盘的渲染位置计算错误**：`PalletLayer.calcWorldPosition` 第 83-96 行只考虑了正向流动的局部坐标计算，对 `_reverseFlow = true` 的托盘没有特殊处理，导致逆流托盘的世界位置可能计算错误。 | `PalletLayer.ts:75-96` | **P1** |
| B8 | **导入场景时不保存撤销快照**：`LoadDialog` 调用 `canvasStore.loadFromJSON` 但未先 `pushUndoSnapshot`，导入后无法 Ctrl+Z 回退。而 Toolbar 中加载场景做了快照（第 33 行）。 | `LoadDialog.vue:13` | **P2** |
| B9 | **批量运行取消信号判断不稳定**：`simulationStore.runBatch()` 第 315 行和第 338 行用 `(this.status as string) === 'idle' && this.multiRunTotal === 0` 判断取消。但 `stop()` 方法设置 `status = 'idle'` 和 `multiRunTotal = 0` 的顺序不保证原子性，可能导致竞态。 | `simulationStore.ts:315, 338` | **P2** |

### 2.2 代码质量

| # | 问题描述 | 位置 | 严重性 |
|---|---------|------|--------|
| Q1 | **console.log 调试日志残留**：`Simulation.ts` 多处 `console.log`（第 116-119, 143, 321-323 行），`simulationStore.ts` 第 100 行，`CanvasContainer.vue` 第 349 行。生产环境应移除或改用条件日志。 | 多处 | **P2** |
| Q2 | **Worker 与 Direct 模式双路径维护成本高**：`simulationStore` 中 Worker 模式和 Direct 回退模式的命令分发逻辑分散在 `start/pause/resume/stop/step/setSpeed/setMaxMode` 等方法中，每个方法都需要 `if (this.directSim)` + `this.sendCommand()` 双写。应抽象为统一接口。 | `simulationStore.ts` | **P2** |
| Q3 | **类型安全不足**：`simulationStore.sendCommand(msg: any)` 参数为 `any`；批量运行中 `(this.status as string)` 强制类型转换；`editorStore` 的 clipboard 中 `Record<string, unknown>` 丢失具体类型信息。 | `simulationStore.ts:208`, 多处 | **P2** |
| Q4 | **SnapManager 重复的 getPortPos 逻辑**：`SnapManager.getPortPos` 与 `ConveyorLayer.getPortPos` 逻辑完全重复，且都硬编码了移载机半宽 `25`。应抽取为共享工具函数。 | `SnapManager.ts:320-336`, `ConveyorLayer.ts:76-97` | **P3** |
| Q5 | **CanvasManager 在 PixiJS 对象中直接持有 Store 引用**：`CanvasManager`、`SelectionManager`、`SnapManager` 等在构造函数中直接 `useCanvasStore()` / `useEditorStore()`，与 Vue 的响应式系统耦合。虽然目前可工作，但违反了关注点分离原则，也不利于单元测试。 | 多处 | **P3** |
| Q6 | **热力图在非仿真状态也每帧更新**：`CanvasManager.onTick` 中即使 `status === 'idle'` 且没有历史数据时，也调用 `heatmapLayer.update({}, [])`，产生不必要的 GPU 绘制。 | `CanvasManager.ts:99-104` | **P3** |

### 2.3 UX 问题

| # | 问题描述 | 严重性 |
|---|---------|--------|
| U1 | **无组件名称/标签系统**：所有组件仅用 ID 后4位标识（如 `a1b2`），用户无法区分同类组件。缺少自定义命名功能。 | **P1** |
| U2 | **右面板宽度固定 240px，属性编辑空间不足**：输送机属性有 5 个输入框 + 移载机有路由表，240px 宽度下输入框过窄，移动端更不可用。 | **P2** |
| U3 | **仿真中完全锁定编辑但无视觉反馈**：仿真运行时编辑锁定仅靠右侧面板一行红色文字提示，画布上的组件没有任何不可编辑的视觉指示。 | **P2** |
| U4 | **键盘快捷键缺乏帮助/提示**：R 旋转、S 连线、Delete 删除、Ctrl+C/V 复制粘贴等快捷键没有任何 UI 提示，新用户难以发现。 | **P2** |
| U5 | **场景库操作无确认**：删除场景只有一个小 `x` 按钮，无二次确认，误触即删。 | **P2** |
| U6 | **数据面板信息密度高但缺少图表可视化**：利用率只有条形进度条，停留时间只有数字，拥堵事件只显示最近5条。缺少趋势图、直方图等更直观的可视化。 | **P2** |
| U7 | **无缩放比例显示**：滚轮缩放后用户无法知道当前缩放比例，也无法精确回到 100%。 | **P3** |
| U8 | **连线工具交互不直观**：需要先选连线工具再点击两个端口，没有拖拽连线支持，也没有端口悬停高亮提示。 | **P2** |
| U9 | **框选必须 Shift+拖拽**：大多数设计工具的框选是空白区域直接拖拽，Shift 用于追加选择更合理。当前逻辑是空白处拖拽 = 平移，对非 CAD 用户不够直观。 | **P3** |
| U10 | **清空按钮无确认**：一键清空画布无任何确认弹窗，误触即丢失所有内容。 | **P1** |

### 2.4 功能缺失

| # | 功能描述 | 影响 | 优先级 |
|---|---------|------|--------|
| F1 | **路由表 action 到端口的映射**：路由表的 `'straight'`/`'turn'` 需要映射为移载机的实际端口名（`north`/`south`/`east`/`west`），目前 B6 标记的 Bug 就是缺失此映射。这是移载机功能的核心依赖。 | **P0** |
| F2 | **仿真事件日志面板**：仿真中的 `PALLET_GENERATED`、`PALLET_CONSUMED`、`CAPACITY_REJECTED` 等事件只通过 toast 通知，无法回溯查看。需要独立的事件日志面板，支持按类型/时间筛选。 | **P1** |
| F3 | **组件标签/命名系统**：用户应为每个组件设置可读名称，在画布和属性面板中显示。 | **P1** |
| F4 | **仿真结果持久化**：目前仿真结果仅存于内存，停止后刷新页面即丢失。应支持导出仿真报告（JSON/CSV），或与场景一起保存分析快照。 | **P2** |
| F5 | **撤销重做完整支持**：目前只有 Ctrl+Z 撤销，缺少 Ctrl+Y / Ctrl+Shift+Z 重做。 | **P2** |
| F6 | **多选组件批量编辑**：框选多个组件后，属性面板应支持批量修改共同属性（如速度、间隔）。 | **P2** |
| F7 | **实时仿真时间显示**：状态栏显示 Tick 和托盘数，但不显示仿真时间（秒/分/时），用户需要看右面板才能知道仿真走了多久。 | **P2** |
| F8 | **拖拽连线**：从端口拖拽到另一个端口直接创建连线，替代当前"选工具 → 点端口A → 点端口B"的交互方式。 | **P3** |
| F9 | **键盘快捷键面板**：可打开的快捷键参考面板。 | **P3** |
| F10 | **方案模板/示例场景**：内置几种典型物流场景模板，降低新用户上手门槛。 | **P3** |
| F11 | **分时段统计**：按时间段（如每分钟/每5分钟）输出吞吐量和利用率变化，支持发现随时间恶化的瓶颈。 | **P2** |
| F12 | **连接验证/错误提示**：不合法连接（如滚筒→链条直接连接）在创建时给出视觉提示，而非静默拒绝。 | **P2** |
| F13 | **仿真结果持久化**：导出仿真报告（JSON/CSV），或将分析快照与场景一起保存。 | **P2** |

## 四、排除项（本期不做）

| 项目 | 原因 |
|------|------|
| 多人协作/云端同步 | 需要后端支持，超出当前项目范围 |
| 移动端适配 | 技术栈基于 Canvas + 键盘操作，移动端体验天然受限 |
| 3D 可视化 | 架构变更大，非当前阶段目标 |
| 自定义组件插件系统 | 复杂度过高，需独立评估 |
| 国际化（i18n） | 当前用户群为中文，暂不需要 |
| 亮色/暗色模式切换 | 深色主题一致性好，CSS 变量已预留扩展基础 |
| CSS 框架（Tailwind/UnoCSS） | 已有完整 scoped CSS 体系，引入框架收益低 |
| Vue Router 路由改造 | 单页应用无需路由 |
